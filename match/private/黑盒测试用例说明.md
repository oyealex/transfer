# 黑盒测试用例说明

## 1. 基本原则

黑盒 API 用例基于 `extra/baseline/` 正确设计文档和 `public/test-cases/API基线文档.md` 冻结 API 契约编写，不依赖实际代码实现。

每个用例必须满足：

1. 可以单独运行。
2. 可以重复运行。
3. 不依赖其他测试创建的数据。
4. 不依赖测试执行顺序。
5. 前置条件通过 REST API 创建。
6. 不直接访问数据库、Repository、Service 或内部类。
7. 不依赖实现细节、日志内容或私有表结构。
8. 用例结束后必须恢复环境，不能留下数据库、缓存、故障注入、测试时钟或事件状态。

## 2. 状态隔离

每个测试用例启动和结束时都应处于干净状态。强制执行：

1. BeforeEach 由测试 harness 启动新的 Spring Boot 上下文。
2. BeforeEach 使用随机 H2 内存库、测试 profile 和干净缓存状态。
3. BeforeEach 由测试 harness seed 管理员账号，并通过 `POST /api/v1/users/login` 获取管理员 token。
4. 每个测试使用唯一 `testRunId`。
5. 所有业务唯一字段带 `testRunId` 后缀。
6. 查询和断言只针对本用例创建的数据。
7. AfterEach 由测试 harness 关闭上下文，丢弃随机 H2 内存库、缓存、故障注入、测试时钟和事件状态。

唯一字段包括：

| 字段 | 示例 |
|------|------|
| email | `user-{testRunId}@example.com` |
| phone | `139{random8}` |
| skuCode | `SKU-{testRunId}` |
| warehouseCode | `WH-{testRunId}` |
| couponCode | `CP-{testRunId}` |
| externalOrderNo | `EO-{testRunId}` |
| clientPaymentNo | `PAY-{testRunId}` |

## 3. 测试支持 REST API

为保证所有 REST 黑盒用例都能通过 HTTP 创建前置条件和观察结果，项目提供以下公开 REST 管理接口：

| Method | URL | 用途 |
|--------|-----|------|
| PUT | `/api/v1/admin/system/configs/{key}` | 覆盖测试配置，如 `invoice.tax-rate`、活动系数、会员等级 |
| GET | `/api/v1/admin/system/configs/{key}` | 查询运行期配置 |
| POST | `/api/v1/admin/ops/fault-injections` | 注入模块故障，如物流创建失败、通知发送失败、订单查询不可用 |
| DELETE | `/api/v1/admin/ops/fault-injections` | 清空所有故障注入 |
| GET | `/api/v1/admin/events/failures` | 查询事件监听失败记录 |
| GET | `/api/v1/admin/notifications` | 查询统一通知发送记录 |
| PUT | `/api/v1/admin/system/clock` | 设置测试时钟，用于超时和过期场景 |
| DELETE | `/api/v1/admin/system/clock` | 重置测试时钟 |
| POST | `/api/v1/admin/orders/timeout-cancel` | 触发订单超时取消扫描 |

以上接口均使用 ADMIN 认证，并纳入黑盒测试 API 契约。数据库和缓存隔离由测试 harness 负责，不依赖被测应用的 reset/bootstrap 业务接口。

## 4. 运行方式

公开和隐藏黑盒用例都是独立 Maven 测试项目，不加入 `public/code/pom.xml` 的 `<modules>`。运行前必须先安装基础工程，使黑盒模块解析到当前源码构建出的 `ecommerce-app`。

从仓库根目录运行公开黑盒：

```bash
mvn -f public/code/pom.xml install
mvn -f public/test-cases/pom.xml test
```

从仓库根目录运行隐藏黑盒：

```bash
mvn -f public/code/pom.xml install
mvn -f private/test-cases-internal/pom.xml test
```

运行单个测试类：

```bash
mvn -f public/test-cases/pom.xml -Dtest=PublicL1Test test
mvn -f private/test-cases-internal/pom.xml -Dtest=HiddenL2Test test
```

不得为了运行黑盒用例修改业务工程 POM；隐藏黑盒模块不得进入选手包。

## 5. 公共 Fixture

公共 Fixture 用于封装 REST API 调用，降低公开和隐藏黑盒用例的重复代码。Fixture 不得访问数据库、Repository、Service 或 Java 内部实现。

| 组件 | 职责 |
|------|------|
| TestRunContext | 生成 testRunId、随机邮箱、SKU 编码、订单外部号 |
| ApiClient | HTTP 调用、Token 管理、错误解析 |
| UserFixture | 注册、激活、登录、创建地址、冻结用户 |
| ProductFixture | 创建 SPU、SKU、上架、商品查询 |
| InventoryFixture | 创建仓库、入库、库存查询 |
| CartFixture | 加购物车、修改数量、价格预估 |
| OrderFixture | 创建订单、查询订单、取消订单、批量订单 |
| PaymentFixture | 发起支付、支付回调、查询支付单 |
| RefundFixture | 申请退款、审核、仓库验收 |
| PromotionFixture | 创建优惠券、领取优惠券、计算优惠 |
| LogisticsFixture | 拣货、打印面单、出库、物流回调 |
| LoyaltyFixture | 查询积分、估算抵扣、处理过期 |
| ReviewFixture | 发布评价、审核评价 |
| InvoiceFixture | 申请发票、查询发票 |

典型方法：

```text
UserFixture.registerAndActivateUser()
UserFixture.createDefaultAddress(userToken)
ProductFixture.createOnShelfSku(adminToken, price)
InventoryFixture.createWarehouseAndInbound(adminToken, skuId, quantity)

OrderFixture.createOrder(userToken, addressId, items, coupons, points)

```

## 8. 静态评分检查

以下检查不属于 REST 黑盒 API 用例。它们需要评分器直接读取提交目录中的文件或扫描 Controller 源码/字节码完成。

| ID | 检查对象 | 检查方式 | 通过条件 |
|----|----------|----------|----------|
| STATIC-DOC-01 | `extra/baseline/` | 文件 hash 或文本 diff | 基线设计副本未被修改 |
| STATIC-DOC-02 | `public/design-docs/附录A-API接口参考.md` | 读取文件文本 | 附录A API URL 未被批量修改 |
| STATIC-DOC-03 | `public/design-docs/02-系统架构.md` | 读取文件文本 | 模块架构和模块边界章节未被删除或重写 |
| STATIC-DOC-04 | `public/design-docs/附录C-数据模型.md` | 读取文件文本 | 核心表和关键字段未被删除 |
| STATIC-CONTRACT-01 | Controller 映射 | 扫描 `@RequestMapping`、`@GetMapping`、`@PostMapping` 等 | 冻结 API 均存在，URL 和 Method 不变 |
| STATIC-CONTRACT-02 | DTO 字段 | 反射或源码扫描 | 关键 Request/Response 字段名和类型兼容 |

REST 黑盒用例只验证通过 API 可观察到的业务行为，例如状态码、响应字段、后续查询结果、事件失败记录、通知查询结果等。文档文本、源码注解、Controller 列表、DTO 字段结构不应伪装成 REST 黑盒用例，应由评分器静态检查。

| L3-08 | order | 超时取消只改订单状态 | 还必须释放库存预占 |
| L3-09 | review/loyalty | 评价提交即发积分 | 审核通过后发积分 |
| L3-10 | invoice/payment | 结算批次包含未支付订单 | 只包含支付成功且未结算订单 |

### L4 模块边界级

| 编号 | 模块 | 错误实现 | 正确设计 |
|------|------|----------|----------|
| L4-01 | payment → order | 直接 JdbcTemplate 查 orders | 通过 OrderQueryService |
| L4-02 | product → inventory | 直接注入 InventoryRepository | 通过 InventoryQueryService |
| L4-03 | review → order | 保存评价前不校验购买 | 调用订单验证购买记录 |
| L4-04 | cart | 购物车落 H2 表 | 使用 Caffeine Cache，TTL 7 天 |
| L4-05 | order → logistics | 支付确认同步创建发货单 | 发布 OrderPaidEvent |
| L4-06 | inventory → product | 库存模块定义 Product Entity | 通过 ProductQueryService |
| L4-07 | all → common | 直接调用 MockMailSender/MockSmsSender | 统一 LocalNotificationService |
| L4-08 | loyalty → order | 直接读取 orders 统计会员等级 | 通过 OrderQueryService 或销售统计接口 |

### L5 架构级

| 编号 | 错误实现 | 正确设计 |
|------|----------|----------|
| L5-01 | PaymentService.confirmPayment 同步执行订单、物流、积分、通知，任一失败导致支付失败 | 支付主流程提交后发布事件，非关键后置动作失败不回滚支付 |
