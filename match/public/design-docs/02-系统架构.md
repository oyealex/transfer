# 系统架构

## 1. 架构风格

ShopHub 采用模块化单体架构。所有模块部署在同一个 Spring Boot 应用中，共享 JVM、事务管理器、H2 数据库连接池和本地事件总线，但每个模块拥有自己的包边界、领域服务、Repository 和对外契约。

模块之间的协作优先级如下：

1. 同步本地接口：用于查询类、校验类、强一致链路。
2. 本地领域事件：用于支付成功后的后置动作、通知、积分、物流等弱耦合链路。
3. REST API：用于黑盒测试和外部客户端访问。
4. 禁止跨模块直接注入对方 Repository 或直接查询对方表。

## 3. 模块边界规则

| 规则 | 要求 |
|------|------|
| 数据访问 | 只能访问本模块拥有的表和 Repository |
| 查询依赖 | 跨模块查询必须通过 QueryService 接口 |
| 命令依赖 | 跨模块命令必须通过领域服务接口或事件 |
| 事件依赖 | 后置动作优先使用 ApplicationEvent |
| DTO | 跨模块传输使用 DTO，不暴露 JPA Entity |
| 事务 | 不允许一个模块的事务依赖非关键后置监听器成功 |

## 4. 关键本地接口

| 接口 | 提供方 | 使用方 | 用途 |
|------|--------|--------|------|
| UserQueryService | user | order、review | 查询用户状态、冻结状态 |
| ProductQueryService | product | inventory、cart、order | 查询商品、SKU、上下架状态 |
| InventoryQueryService | inventory | product、cart、order | 查询库存摘要 |
| InventoryReservationService | inventory | order | 预占、释放、扣减库存 |
| OrderQueryService | order | payment、review、logistics、loyalty | 查询订单、验证购买记录、提供会员等级统计所需订单数据 |
| OrderPaymentStatusUpdater | order | payment | 更新订单支付状态 |
| PromotionCalculationService | promotion | order、cart | 计算优惠 |
| LogisticsCommandService | logistics | event listener | 创建发货单 |
| LoyaltyCommandService | loyalty | order、payment | 积分抵扣、发放 |
| LocalNotificationService | common | all | 统一通知发送 |

## 5. 事件驱动设计

核心事件：

| 事件 | 发布方 | 监听方 | 失败策略 |
|------|--------|--------|----------|
| UserRegisteredEvent | user | common notification | 失败记录日志，不回滚注册 |
| OrderCreatedEvent | order | notification | 失败记录日志，不回滚订单 |
| OrderPaidEvent | order | logistics、loyalty、notification | 失败记录补偿任务，不回滚支付 |
| PaymentSucceededEvent | payment | order、logistics、loyalty、notification | 订单状态更新失败应告警，非关键后置动作不回滚支付 |
| ShipmentDeliveredEvent | logistics | order、loyalty | 更新订单签收状态，失败可重试 |
| ReviewApprovedEvent | review | loyalty | 发放评价积分 |
| RefundCompletedEvent | payment | order、notification | 更新售后状态、通知用户 |

## 6. 事务边界

1. 创建订单事务只包含订单主数据、订单明细、库存预占记录和优惠使用记录。
2. 支付确认事务只包含支付单状态、订单支付状态和库存扣减。
3. 支付后物流、积分、通知通过事件异步处理，不得使支付确认失败。
4. 批量订单导入按单条事务处理，一条失败不得回滚整批。
5. 退款流程按售后单、验收单、退款单分阶段提交。

## 7. 缓存设计

| 缓存 | Key | TTL | 所属模块 |
|------|-----|-----|----------|
| 购物车 | `cart:{userId}` | 7 天 | cart |
| 商品详情 | `product:detail:{skuId}` | 10 分钟 | product |
| 库存摘要 | `inventory:summary:{skuId}` | 30 秒 | inventory |
| 用户权限 | `user:roles:{userId}` | 30 分钟 | user |
| 运费模板 | `logistics:freight:{templateId}` | 30 分钟 | logistics |

购物车不得落库保存临时明细。订单提交后，订单模块保存订单明细。

## 8. 安全架构

1. 登录成功后签发 JWT。
2. 用户侧接口需要 `Authorization: Bearer <token>`。
3. 管理类接口需要 `ADMIN` 角色。
4. 支付回调接口使用本地模拟签名头 `X-Payment-Signature`。
5. 黑盒测试通过公开 REST 管理接口完成环境重置、管理员初始化、配置覆盖、故障注入和可观察结果查询。
