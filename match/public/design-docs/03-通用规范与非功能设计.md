# 通用规范与非功能设计

## 1. 金额计算

所有金额使用 `BigDecimal`，禁止使用 `double` 或 `float` 表示金额。

金额计算统一规则：

| 场景 | 规则 |
|------|------|
| 中间计算 | 保留足够精度，不提前截断 |
| 最终入库 | 保留 2 位小数 |
| 舍入模式 | `RoundingMode.HALF_UP` |
| 税额 | 税额单独四舍五入到分 |
| 优惠金额 | 不得小于 0，不得大于商品金额 |
| 应付金额 | 不得小于 0.01，0 元订单不在本系统支持范围内 |

## 2. 通用异常

| 异常 | 说明 | HTTP 状态码 |
|------|------|-------------|
| BusinessException | 通用业务异常 | 400 |
| ResourceNotFoundException | 资源不存在 | 404 |
| AuthorizationException | 未认证或无权限 | 401/403 |
| ValidationException | 参数校验失败 | 400 |
| ConflictException | 状态冲突、重复提交 | 409 |
| RateLimitException | 触发限流 | 429 |

订单金额校验失败必须抛出 `OrderValidationException`，不得抛出 Java 标准 `IllegalArgumentException`。

## 3. 幂等规范

以下接口必须支持幂等：

| 接口 | 幂等键 |
|------|--------|
| 创建订单 | `externalOrderNo` |
| 支付回调 | `paymentNo` + `callbackSequence` |
| 退款申请 | `refundRequestNo` |
| 物流回调 | `trackingNo` + `eventTime` + `status` |
| 发票申请 | `invoiceRequestNo` |

重复请求应返回第一次处理结果，不得重复扣款、重复扣库存、重复发积分或重复开票。

## 4. 本地限流

| 接口类别 | 限流规则 |
|----------|----------|
| 登录 | 同一用户名每分钟 5 次 |
| 支付回调 | 同一 paymentNo 每分钟 20 次 |
| 商品搜索 | 同一 IP 每分钟 120 次 |
| 创建订单 | 同一用户每分钟 20 次 |

限流触发时返回 429，错误码为 `RATE_LIMITED`。

## 5. 黑盒测试隔离

黑盒测试隔离由测试 harness 提供，不通过业务 REST API 清理环境。每个测试方法应使用：

```text
新的 Spring Boot 上下文
随机 H2 内存数据库
干净缓存状态
测试 harness seed 的管理员账号
```

业务代码不得依赖或暴露 reset/bootstrap 接口来满足黑盒用例隔离。用例需要创建前置条件时，必须通过正式 REST API 或已纳入契约的 ADMIN 测试支撑接口完成。

## 6. 审计日志

以下操作必须记录审计日志：

1. 用户冻结和解冻。
2. 商品上下架。
3. 库存人工调整。
4. 订单取消审核。
5. 退款审核和仓库验收。
6. 发票开具。
7. 结算批次生成。

审计日志至少包含操作者、操作类型、业务 ID、操作前状态、操作后状态、操作时间和备注。

## 7. 通知规范

所有通知必须通过 `LocalNotificationService` 发送。业务模块只能提交 `NotificationRequest`，不得直接调用 `MockMailSender` 或 `MockSmsSender`。

通知组件负责：

1. 模板渲染。
2. 幂等去重。
3. 发送日志。
4. 失败记录。
5. 邮件和短信模拟器适配。

## 8. 本地事件失败处理

事件监听器失败时：

1. 记录事件处理失败日志。
2. 保存失败记录到本地事件处理表。
3. 不回滚主业务事务，除非该监听器是明确声明的强一致监听器。
4. 可通过管理接口重放失败事件。

支付成功后的物流、积分、通知监听器均为非强一致监听器。
