# ShopHub 项目概述

## 1. 系统定位

ShopHub 是一个面向 B2C 场景的综合电商平台，覆盖从用户注册、商品浏览、购物车、下单、支付、库存履约、物流配送、售后退款、评价、积分会员、发票结算到销售统计的完整业务链路。

系统以模块化单体方式交付。所有模块运行在同一个 Spring Boot 应用进程内，但按照清晰的领域边界组织代码、数据访问和事件交互。模块之间不得随意访问彼此的数据库表或 Repository，应通过公开的本地接口、REST API、领域服务或 Spring ApplicationEvent 完成协作。

## 2. 技术栈

| 类型 | 选型 |
|------|------|
| 语言 | Java 17 |
| 框架 | Spring Boot 3.2.x |
| 构建 | Maven 多模块 |
| ORM | Spring Data JPA / Hibernate |
| 数据库 | H2 文件模式，测试使用 H2 内存模式 |
| 缓存 | Spring Cache + Caffeine |
| 事件 | Spring ApplicationEvent |
| 认证 | Spring Security + JWT |
| 限流 | 单机本地限流器 |
| 测试 | JUnit 5、Mockito、MockMvc、TestRestTemplate |

系统必须能够在独立主机上直接运行，不依赖外部数据库、Redis、消息队列、搜索引擎、支付网关、短信网关或第三方物流接口。

## 3. API 版本约定

所有 REST API 统一使用 `/api/v1/` 作为接口前缀。当前版本为 v1。

除黑盒测试支撑管理接口外，所有业务 API 必须在附录A中登记。代码实现不得修改附录A中定义的 URL、HTTP Method、请求字段、响应字段和错误响应结构。

## 5. 关键业务原则

1. 商品下单前必须校验用户状态、商品状态、库存可用性、促销有效性和风控结果。
2. 购物车是临时数据，存储在 Caffeine 本地缓存中，TTL 为 7 天。
3. 创建订单时只预占库存，不扣减库存。支付成功后扣减库存。
4. 支付金额必须等于订单应付金额，本系统不支持部分支付。
5. 支付成功后的物流创建、积分发放、通知发送等后置动作通过本地事件异步触发，不得阻塞支付主流程。
6. 已支付订单取消必须进入商家审核流程，审核通过后按退款流程处理。
7. 退款必须经过商家审核和仓库验收，验收通过后才可退款。
8. 发票支持部分开票，一个订单可开多张发票，累计开票金额不得超过订单实付金额。
9. 积分抵扣同时受单笔 10,000 积分上限和订单实付金额 50% 上限约束。
10. 评价必须校验用户已购买且订单已签收。

## 6. 统一数据格式

| 类型 | 规则 |
|------|------|
| 金额 | 使用 `BigDecimal`，接口中以数字传输，保留 2 位小数 |
| 时间 | 使用 ISO-8601 字符串，时区为系统默认时区 |
| ID | 数据库主键使用 Long，接口中使用 number |
| 布尔值 | 使用 boolean |
| 枚举 | 使用大写英文字符串 |
| 分页 | 使用 `page`、`size`、`total`、`items` |

## 7. 统一响应格式

成功响应直接返回业务 DTO。分页响应返回：

```json
{
  "page": 0,
  "size": 20,
  "total": 100,
  "items": []
}
```

错误响应统一返回：

```json
{
  "code": "ORDER_INVALID_AMOUNT",
  "message": "Order amount must be greater than zero",
  "traceId": "20260607-abcdef",
  "details": {}
}
```

## 8. 运行模式

| Profile | 说明 |
|---------|------|
| default | 本地开发模式，H2 文件数据库，启用本地缓存 |
| test | 测试模式，H2 内存数据库，由测试 harness 提供用例级隔离 |
| prod-like | 单机演示模式，使用文件数据库和固定 JWT 密钥 |

黑盒测试通过测试 harness 在每个测试方法中启动新的 Spring Boot 上下文、随机 H2 内存库和干净缓存状态，并 seed 管理员账号后通过登录接口获取管理员 token。用例隔离不依赖业务 reset/bootstrap 接口。

黑盒测试需要的可观察支撑能力通过 `/api/v1/admin/` 下的 REST 管理接口提供，包括配置覆盖、故障注入、测试时钟、事件失败记录和通知记录查询。这些接口均使用 ADMIN 认证，并纳入 API 契约。
