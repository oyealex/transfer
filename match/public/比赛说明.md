# 比赛说明

本文档中的路径均以本文档所在目录为根目录。

## 1. 比赛主题

设计实现一致性检查与修复。

参赛者需要使用 AI Agent 检查 ShopHub 电商系统的设计文档与代码实现，发现不一致点，并修复代码。

## 2. 任务目标

1. 阅读 `design-docs/` 中的所有设计文档。
2. 阅读 `code/` 中的 Java Spring Boot 项目。
3. 阅读 `test-cases/API基线文档.md` 中冻结的 REST API 契约。
4. 对比设计文档和代码实现。
5. 找出普通不一致点并修复代码。
6. 保持项目可编译。
7. 输出修复报告。

## 3. 项目结构

```text
public/
├── code/                 # 基础工程代码，含与当前基础代码自洽的单元测试
├── design-docs/          # 业务设计文档，最终验收以此为准
├── test-cases/           # 公开黑盒 JUnit 项目、API 基线和公开用例说明
└── 比赛说明.md           # 项目背景、任务规则、提交要求和验证方式
```

`test-cases/` 中文档说明：

| 文档 | 内容 |
|------|------|
| `test-cases/API基线文档.md` | 冻结 REST API 签名、接口索引、错误码和黑盒支撑管理接口 |
| `test-cases/黑盒用例说明.md` | 公开黑盒用例运行方式、隔离规则、用例清单和 Given-When-Then 规格 |

## 4. 当前代码现状

`code/` 是一个多模块 Spring Boot 电商系统。代码可以编译，模块内单元测试与当前基础代码自洽；但当前实现故意包含若干与设计文档不一致的问题，公开黑盒用例在初始代码上不一定全部通过。

## 5. 允许修改

| 类型 | 是否允许 |
|------|----------|
| Java 源代码 | 允许 |
| application.yml | 允许 |
| pom.xml | 允许 |
| JUnit 测试 | 允许修改或删除 |

## 6. 禁止修改

1. REST API URL。
2. HTTP Method。
3. Request Header 定义。
4. Request Body 字段名和类型。
5. Response Body 字段名和类型。
6. 设计文档章节。
7. 为隐藏测试硬编码逻辑。
8. `design-docs/` 中的设计文档。

## 7. 如何验证

运行基础工程单元测试：

```bash
cd public
mvn -f code/pom.xml test
```

安装基础工程，供独立黑盒测试模块解析依赖：

```bash
mvn -f code/pom.xml install
```

运行公开黑盒用例：

```bash
mvn -f test-cases/pom.xml test
```

公开黑盒用例是独立 Maven 测试项目，不加入 `code/pom.xml` 的 `<modules>`。运行公开黑盒前必须先安装基础工程；不需要也不应为了运行黑盒用例修改业务工程 POM。

## 9. Agent 工作步骤

1. 通读 `design-docs/`。
2. 阅读 `code/` 中的模块结构和核心实现。
3. 阅读 `test-cases/API基线文档.md`，确认冻结 API 签名。
4. 运行 `test-cases/` 公开黑盒 API 用例，观察失败场景。
5. 按模块对照设计文档和代码实现。
6. 修复代码中与设计不一致的问题。
7. 输出完整修复报告。

设计文档是验收基准，代码需要匹配设计。不要因为代码当前行为或公开测试现状去修改设计文档。

## 10. 修复报告

修复报告必须包含：

1. 不一致点清单。
2. 每个问题的设计依据。
3. 每个问题的实际代码行为。
4. 修复方式。
5. 未修复风险说明。

报告建议使用以下结构：

```markdown
## 发现的不一致点

| 编号 | 严重程度 | 位置 | 设计描述 | 代码行为 | 类型 | 是否修复 |
|------|----------|------|----------|----------|------|----------|

## 修复详情

逐项说明修改文件、修改前行为、修改后行为和验证方式。

## 总结

发现总数：
已修复数量：
仍有风险：
```

## 11. FAQ

### 是否可以修改设计文档？

不可以。设计文档是验收基准，参赛者应修改代码以匹配设计。

### 是否可以修改 JUnit？

可以。JUnit 不参与最终评分。最终评分基于隐藏黑盒 API 用例和报告。

### 是否可以修改 API URL？

不可以。REST API 签名被冻结，URL、Method、Request、Response 都不得修改。

### 是否允许新增类？

允许。可以新增服务、配置、事件、DTO、测试辅助类等，但不得破坏 API 契约。
